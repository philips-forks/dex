## Summary

The [Enterprise-Managed Authorization (EMA)][ema-spec] extension
(`io.modelcontextprotocol/enterprise-managed-authorization`) lets an
organization control MCP server access centrally through its enterprise IdP.
EMA is **not a new protocol** — it is an MCP profile over four existing OAuth
specs:

- Token Exchange [RFC 8693] — obtain the ID-JAG from the IdP.
- [Identity Assertion JWT Authorization Grant (ID-JAG)][id-jag] — the brokered
  assertion the IdP mints.
- JWT Profile for OAuth 2.0 Authorization Grants [RFC 7523] — redeem the ID-JAG
  for an access token at the MCP server's authorization server.
- Protected Resource Metadata [RFC 9728] — how the client discovers that
  authorization server.

EMA defines a three-party flow with **two OAuth server roles**, and Dex can
play either or both:

- **Role A — Enterprise IdP**: mints ID-JAGs after evaluating org policy.
  **This is already designed in [DEP 4600][dep4600]**.
- **Role B — MCP Authorization Server (MCP AS)**: validates an ID-JAG minted by
  an upstream IdP and issues an MCP access token. **DEP 4600 lists this as a
  non-goal / future improvement** — this DEP designs it, plus the EMA-specific
  discovery and resource-binding constraints.

[ema-spec]: https://modelcontextprotocol.io/extensions/auth/enterprise-managed-authorization
[id-jag]: https://datatracker.ietf.org/doc/draft-ietf-oauth-identity-assertion-authz-grant/
[dep4600]: ./id-jag-2026-03-02%234600.md
[RFC 8693]: https://datatracker.ietf.org/doc/html/rfc8693
[RFC 7523]: https://datatracker.ietf.org/doc/html/rfc7523
[RFC 9728]: https://datatracker.ietf.org/doc/html/rfc9728
[RFC 8707]: https://datatracker.ietf.org/doc/html/rfc8707

## Context

- [DEP 4600 - ID-JAG][dep4600] designs Dex as the **IdP** that issues ID-JAGs via
  Token Exchange (`requested_token_type=urn:ietf:params:oauth:token-type:id-jag`),
  including static per-client `idJAGPolicies`, the `oauth-id-jag+jwt` JWT shape,
  and discovery metadata. EMA's Role A is satisfied by DEP 4600 essentially
  unchanged.
- [DEP #2812 - RFC 8693 Token Exchange](https://github.com/dexidp/dex/pull/2812)
  established the Token Exchange foundation both DEPs build on.
- The [EMA spec][ema-spec] is **Stable** in the MCP `ext-auth` repository and is
  a profile of the IETF ID-JAG draft (`-04`).

The EMA core flow (spec §2):

```
 MCP Client --(SSO / OIDC)-->        Enterprise IdP   : login -> ID Token
 MCP Client --(RFC 8693 exchange)--> Enterprise IdP   : ID Token -> ID-JAG     (Role A)
 MCP Client --(RFC 7523 jwt-bearer)->MCP AS           : ID-JAG  -> access tok  (Role B)
 MCP Client --(Bearer token)-------> MCP Resource Srv : call MCP API
```

## Motivation

### Goals/Pain

In enterprise MCP deployments, the default per-user/per-server authorization
model creates friction and security gaps: every employee must individually
authorize every MCP server, security teams cannot enforce consistent policy,
and on/offboarding requires touching each service. EMA moves the access
decision to the enterprise IdP so that:

- Employees use existing corporate SSO; no per-server consent prompts.
- Admins gain central visibility and policy control over which clients may use
  which MCP servers, and with which scopes.
- Revocation happens once, at the IdP, and takes effect across all clients.

Concrete goals for Dex:

- **Role A (have):** issue EMA-conformant ID-JAGs. Covered by DEP 4600; this DEP
  only adds the EMA profile constraints on top.
- **Role B (new):** accept an ID-JAG via `grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer`,
  validate it against a trusted IdP's JWKS, and issue an MCP access token
  **audience-restricted to the MCP server**.
- **Discovery (new):** advertise EMA support so MCP clients use the brokered flow
  instead of redirecting the user to Dex's authorization endpoint.

### Non-goals

- A general-purpose policy / conditional-access engine (MFA step-up, device
  posture). Coarse static allow-lists only (consistent with DEP 4600).
- An IdP admin console or "resource descriptor" publishing API for admins to
  register MCP servers — that is MCP-server/IdP-product territory.
- SAML assertion → refresh token exchange branch (EMA §4 SAML path). OIDC
  ID-token path only. Consistent with DEP 4600 deferring SAML.
- Client ID Metadata Documents (`client_id` as a URL) and `private_key_jwt`
  client auth (EMA §5). Dex's DCR is the registration story; CIMD is a separate
  optional mechanism, deferred.
- MCP transport concerns (`initialize` capability negotiation) — entirely the
  MCP client/server layer.

## What EMA actually is

EMA layers MCP-specific constraints onto the generic ID-JAG draft. The pieces
that matter for Dex:

| EMA element | Spec ref | Meaning for Dex |
|---|---|---|
| Token Exchange to mint ID-JAG | §4 | `audience` MUST be the **issuer identifier of the MCP AS**; `resource` OPTIONAL, MUST be the MCP server's Resource Identifier (RFC 9728) if present |
| ID-JAG JWT | §4.3 | `typ: oauth-id-jag+jwt`; claims `jti, iss, sub, email?, aud, resource?, client_id, exp, iat, scope`. `resource` MUST be the MCP server's Resource Identifier if present |
| Access token request | §5 | `grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer`, `assertion=<ID-JAG>` |
| Access token | §5.1 | Issued token **MUST be audience-restricted** to the MCP server named by the ID-JAG `resource` claim |
| Discovery | §6 | MCP AS metadata advertises `urn:ietf:params:oauth:grant-profile:id-jag` in `authorization_grant_profiles_supported` |

## Where Dex stands today

Verified against the current tree:

| Capability | Status | Location |
|---|---|---|
| RFC 8693 token-exchange grant | present | `server/handlers.go` `handleTokenExchange` |
| `subject_token_type` id_token/access_token | present (only these two) | `server/handlers.go` |
| ID-JAG issuance (`requested_token_type=...id-jag`, `oauth-id-jag+jwt`) | **designed in DEP 4600**, not yet merged | n/a |
| `jwt-bearer` grant (RFC 7523) | **absent** | not in `allSupportedGrants` (`server/server.go`), not in token switch |
| Audience-restrict access token to a `resource` | **absent** | `getAudience` (`server/oauth2.go`) always binds to client / cross-client peers |
| `resource` request param (RFC 8707) | **absent** | not parsed at `/token` |
| Protected Resource Metadata (RFC 9728) | **absent** | no `/.well-known/oauth-protected-resource` |
| `authorization_grant_profiles_supported` in AS metadata | **absent** | `discovery` struct (`server/handlers.go`) |
| External JWKS verification of another IdP's tokens | partial / reusable | connector `TokenIdentity` verifies upstream id_tokens |

Net: Dex has the JWT signing and token-exchange plumbing. DEP 4600 fills Role A.
The remaining EMA gaps are **Role B (jwt-bearer redemption + resource-audience
binding)** and **EMA discovery (RFC 9728 + grant-profile advertisement)**.

## Proposal

### Role A — Dex as Enterprise IdP (covered by DEP 4600)

EMA's Role A is the ID-JAG issuance flow already specified in [DEP 4600][dep4600].
EMA adds only profile constraints, which DEP 4600 already satisfies or trivially
extends:

1. **`audience` = MCP AS issuer** (EMA §4). DEP 4600 already requires `audience`
   as the Resource AS issuer URL. No change.
2. **`resource` = MCP server Resource Identifier** (EMA §4, §4.3). DEP 4600
   already accepts the OPTIONAL `resource` param and is to place it in the
   ID-JAG. EMA requires that, if present, it be the RFC 9728 Resource
   Identifier — a documentation/validation refinement, not new code.
3. **ID-JAG shape** (`typ: oauth-id-jag+jwt`, `email` claim). DEP 4600 already
   mints this; EMA's §"For MCP Authorization Servers" notes `email` MAY be
   present for account linking — ensure DEP 4600 carries the `email` claim when
   the connector provides a verified email.
4. **Static policy** (`idJAGPolicies` allow-list per client). Matches the chosen
   static allow-list approach; no new policy engine.

**Action for EMA:** verify DEP 4600's ID-JAG carries `email`, and document the
EMA profile (`resource` = RFC 9728 identifier) in the DEP 4600 config docs. No
separate Role A implementation is needed.

### Role B — Dex as MCP Authorization Server (new)

This is the net-new work. It lets Dex front MCP servers that trust ID-JAGs
minted by an upstream enterprise IdP (Okta/Entra/… or another Dex running
Role A).

1. **Add the `jwt-bearer` grant.**
   - New constant `grantTypeJWTBearer = "urn:ietf:params:oauth:grant-type:jwt-bearer"`
     in `server/oauth2.go`.
   - Register it in `allSupportedGrants` (`server/server.go`) **only when EMA is
     configured** (see config) and behind a `featureflags` flag so default
     behavior is unchanged.
   - Add a `case grantTypeJWTBearer:` to the token switch (`server/handlers.go`)
     dispatching to a new `handleJWTBearerGrant`.

2. **Trusted-issuer ID-JAG validation** (the crux).
   - New config block listing trusted IdP issuers: `issuer`, `jwksURL` (or reuse
     an OIDC connector's keyset), and the expected `audience` (this Dex's issuer).
   - `handleJWTBearerGrant` reads `assertion`, parses the JWT, and validates:
     - signature against the trusted issuer's cached JWKS (reuse `go-oidc`
       remote keyset, same caching pattern connectors use);
     - header `typ == oauth-id-jag+jwt`;
     - `iss` is a configured trusted issuer;
     - `aud` == this Dex's issuer URL (EMA §4: audience is the MCP AS issuer);
     - `exp` / `iat` within skew;
     - extracts `sub`, `email`, `scope`, `resource`, `client_id`.

3. **Resource-audience binding** (EMA §5.1 normative MUST).
   - The issued access token's `aud` MUST be the ID-JAG `resource` claim, not the
     requesting client. Today `getAudience` (`server/oauth2.go`) always binds to
     the client / cross-client peers. Add a dedicated path that sets the audience
     to `resource` for this grant, so normal flows do not regress.
   - If `resource` is absent, fall back to `aud` (the MCP AS) per EMA, or reject —
     see Open Questions.

4. **Claim → identity mapping / account linking** (EMA §"For MCP AS").
   - Map ID-JAG `sub` (and optionally `email`) to a Dex `storage.Claims`. v1:
     trust `sub` as-is and copy `email`/`scope` through. Email-based linking is
     deployment-specific and left configurable/off by default.

5. **Scope handling.** Intersect requested/ID-JAG scopes with what the MCP server
   (client registration) allows; emit reduced-scope responses with an explicit
   `scope` field, mirroring DEP 4600's scope-modification behavior.

### Discovery and Protected Resource Metadata

1. **Authorization-server metadata** (EMA §6): add
   `authorization_grant_profiles_supported` to the `discovery` struct
   (`server/handlers.go`) and include
   `urn:ietf:params:oauth:grant-profile:id-jag` when Role B is enabled. (Note
   DEP 4600 adds `identity_chaining_requested_token_types_supported` for Role A;
   these are complementary.)

2. **Protected Resource Metadata** (RFC 9728): add a
   `/.well-known/oauth-protected-resource` handler so MCP clients can discover
   which authorization server (Dex) protects a given MCP server, and the
   `resource` identifier. This is broadly useful beyond EMA and is a prerequisite
   for the MCP client to even find Dex. **May be partly out of scope** — see below.

3. **RFC 8707 `resource` parameter** plumbing at `/token` so non-EMA flows can
   also request resource-bound tokens. Useful groundwork, optional for EMA itself.

### Configuration

Role B introduces a trusted-issuer block. Role A reuses DEP 4600's
`idJAGPolicies`. Example for Dex acting as MCP AS (Role B):

```yaml
oauth2:
  grantTypes:
    - urn:ietf:params:oauth:grant-type:jwt-bearer   # enables Role B
  # EMA / ID-JAG acceptance config
  enterpriseManagedAuthorization:
    enabled: true
    # IdPs whose ID-JAGs Dex will accept and redeem for MCP access tokens.
    trustedIssuers:
      - issuer: "https://acme.okta.example"
        jwksURL: "https://acme.okta.example/oauth2/v1/keys"
        # Dex's own issuer; the ID-JAG `aud` must equal this.
        expectedAudience: "https://mcp-as.example/"
        # Optional: restrict which ID-JAG client_id values are accepted.
        allowedClientIDs: ["f53f191f9311af35"]
    # If true, link ID-JAG identities to local accounts by verified email.
    accountLinkingByEmail: false
```

For Dex acting as Enterprise IdP (Role A), use DEP 4600's config unchanged
(`oauth2.tokenExchange.tokenTypes` + per-client `idJAGPolicies`). For the full
Dex-to-Dex triangle, one Dex runs Role A config and another runs Role B config,
with Role B's `trustedIssuers[].issuer` pointing at Role A's issuer URL.

## Upstream / external requirements

- **Dex as Role B requires the upstream IdP to mint ID-JAGs.** Today, in
  practice, only Okta (Cross-App Access) ships this; Entra/others do not yet.
  This is a **hard external dependency**: Dex-as-MCP-AS is inert until the
  customer's IdP supports the ID-JAG draft. The Dex-to-Dex path (Role A + Role B
  both Dex) removes this dependency for testing/demos.
- **Dex as Role A requires the downstream MCP AS to support `jwt-bearer`.** If
  that downstream is also Dex (Role B), it composes. If third-party, no extra
  Dex work.
- **MCP clients** must implement: SSO + store ID token, the RFC 8693 exchange,
  and RFC 7523 redemption, plus declare the EMA capability in `initialize`. This
  is entirely client-side and **not Dex's concern**, but end-to-end testing is
  blocked without a conformant client.

## Out of scope for Dex

1. **Org policy admin console / "resource descriptor" publishing** (EMA "For MCP
   servers" → "Integrate with IdP admin APIs"). IdP-product territory; Dex offers
   only static allow-lists.
2. **SAML assertion → refresh-token exchange branch** (EMA §4 SAML path). Dex's
   SAML connector is inbound-only; OIDC ID-token path only for v1.
3. **Client ID Metadata Documents + `private_key_jwt`** (EMA §5). DCR is the
   registration story; CIMD deferred.
4. **General policy / conditional-access engine** (MFA step-up, device posture).
   The IdP's job; Dex does coarse group/scope gating only.
5. **MCP transport / `initialize` capability negotiation.** MCP client/server
   layer, never Dex.

## Sequencing

1. **Phase 1 — Role B (new, highest near-term value):** `jwt-bearer` grant +
   trusted-issuer JWKS validation + resource-audience binding + the
   `authorization_grant_profiles_supported` discovery field. Behind a feature
   flag. Lets Dex front MCP servers behind a real enterprise IdP (Okta).
2. **Phase 2 — RFC 9728 PRM endpoint + RFC 8707 `resource` param.** Broadly
   useful beyond EMA; required for MCP client discovery.
3. **Phase 3 — Role A profile polish on top of DEP 4600** (`email` claim,
   `resource` = RFC 9728 identifier docs). Small once DEP 4600 lands.
4. **Phase 4 — full Dex-to-Dex triangle integration test** proving Role A + Role
   B compose end-to-end without a third-party IdP.

## Risks and Mitigations

- **External-IdP dependency (Role B):** mitigated by also shipping Role A so the
  Dex-to-Dex path is self-contained for testing and for orgs that run Dex as
  their IdP.
- **Audience-binding regressions:** the `resource`→`aud` change is the riskiest
  edit to shared token code. Isolate it to the jwt-bearer path; add tests
  asserting normal/authorization_code/token-exchange audiences are unchanged.
- **ID-JAG replay:** stolen ID-JAG replayable within its short lifetime. Mitigate
  with short `exp` acceptance window and optional `jti` cache (deferred, matches
  DEP 4600).
- **Trust misconfiguration:** accepting an over-broad `trustedIssuers` entry is a
  full impersonation risk. Default-deny; require explicit `expectedAudience`;
  log every accepted/rejected redemption with `iss`, `client_id`, `resource`.
- **Spec maturity:** EMA is "Stable" in MCP but profiles an IETF *draft* (`-04`);
  claim/field names may shift. Keep behind a feature flag.

## Open Questions

- When the ID-JAG has no `resource` claim, should Role B bind the access token
  `aud` to the ID-JAG `aud` (the MCP AS), or reject? EMA §5.1 implies a
  resource-bound token is required.
- Should `trustedIssuers` reuse the existing OIDC connector machinery (which
  already fetches/caches JWKS and verifies issuers) rather than a parallel
  config block? Reuse reduces code but couples acceptance policy to connector
  config.
- Account linking: trust ID-JAG `sub` opaque, or resolve to a local user via
  `email`? Default off; needs a deployment decision.
- Should Role B be a new top-level config block, or fold into DEP 4600's
  `enterpriseManagedAuthorization`/token-exchange config for a unified surface?
